# AI Context

## Purpose

This document defines how AI assistants should reason about the ReactEvents project.

It does not define:

* UI architecture
* design direction
* spacing systems
* layout rules

Those concerns are governed by their respective architecture documents.

---

# Project Nature

ReactEvents is a React + Next.js learning application.

The primary goal of the project is to learn:

* React architecture
* component design
* frontend design systems
* layout composition
* responsive design
* maintainable code organization
* frontend design principles
* architectural decision making

The project is intentionally used as a learning environment for exploring:

* design patterns
* architectural patterns
* UI composition
* design systems
* refactoring strategies
* frontend maintainability

---

# Technology Stack

The application is built with:

* Next.js App Router
* React
* TypeScript
* Tailwind CSS
* feature-first frontend architecture

* enterprise CRUD systems
* dashboard interfaces

# AI Reasoning Preference

When reviewing code or suggesting improvements, prioritize:

* architectural clarity
* ownership clarity
* component responsibility
* maintainability
* readability
* consistency
* compositional thinking

Prefer reasoning from:

* React architecture
* frontend architecture
* design systems
* UI composition
* responsive design

Avoid defaulting to:

* business strategy
* conversion optimization
* marketing recommendations
* growth-focused suggestions

unless explicitly requested.

---

# Refactor Philosophy

Refactors should:

* preserve existing behavior
* improve clarity
* improve maintainability
* reduce duplication
* strengthen ownership boundaries

Prefer:

* incremental improvements
* local refactors
* semantic abstractions
* constrained reusable patterns

Avoid:

* broad rewrites
* speculative abstractions
* unnecessary complexity
* architecture for hypothetical future needs

---

# Collaboration Rules

When assisting:

* explain architectural reasoning
* explain tradeoffs
* teach patterns when useful
* prefer clarity over cleverness
* prefer maintainability over flexibility

When uncertain:

* choose the simpler solution
* preserve existing direction
* favor explicit ownership
* favor constrained systems

The goal is to learn good architecture and design principles, not maximize abstraction.
